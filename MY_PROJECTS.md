### 4. Technical Challenges & Complex Workflows Resolved
1. **Cumulative export deduplication (`storage.py: _discover_latest_blobs`)** — Azure Cost Management daily exports are cumulative within a billing period: each run writes a new file into the same `YYYYMMDD-YYYYMMDD` folder, so naively loading all blobs would multiply every cost row. The solution lexicographically sorts date-range folder segments, walks newest-first, and selects only the single blob with the latest `last_modified` timestamp within the chosen folder. Falls back to newest `.parquet`/`.csv` if no date-range folder structure is detected. No external coordination required.

2. **Multi-source cost enrichment with 6-tier projection logic (`live_resources.py: _get_live_data`)** — Each live ARM resource gets `monthly_cost` from one of six sources with explicit priority: (a) `spot_rate` — live Azure Retail Prices API filtered for `skuName contains "Spot"`; (b) `price_table` — on-demand retail × 24 × 30 × count; (c) `export_hours` — for VMs with <28 days in export, derives hourly rate from `C_QUANTITY` then extrapolates; (d) `export_projected` — linear extrapolation from partial-month export; (e) `export_last_day` — for Premium File Shares picks higher of window-projected vs last-day-projected; (f) `export` — full 30-day window as-is. Storage accounts whose cost is broken out via file share inventory are zeroed out (`in_shares`) to prevent double-counting. Resource IDs normalised via `_normalize_rid()` regex before DataFrame join.

3. **RBAC via ACA Easy Auth `X-MS-CLIENT-PRINCIPAL` header (`main.py: _get_user_roles`, `_require_admin`)** — `_get_user_roles` decodes the Base64 header, parses `claims[].val` where `typ == "roles"`, yielding Entra app role values. `_require_admin` FastAPI dependency gates all resource mutation endpoints (`DELETE /api/resource`, `/api/resource-group`, `/api/resource-groups/all`). When header is absent (local dev), all access is allowed through — intentional dev convenience that creates a bypass risk if Easy Auth is ever misconfigured.

### 5. Potential Bottlenecks & Technical Debt
- **Single-replica, no HA** — `max_replicas = 1` hardcoded; no horizontal scaling or blue/green. `min_replicas = 0` adds ~30 s cold-start (including ~500 MB Python image pull) on first request after idle.
- **Sequential ARM SDK calls blocking thread pool** — ARM inventory makes one `resources.list()` + N individual `virtual_machines.get(expand="instanceView")` + NIC lookups, all sequential in a single `run_in_executor` task. O(N) latency for large subscriptions.
- **Unbounded pandas in-memory footprint** — Full cost export DataFrame held in a process-level dict with no size limit. Azure CM exports for large subscriptions can reach hundreds of MB; pandas overhead (3–5× raw size) risks OOM at 1 Gi container limit.
- **No test suite** — CI runs only ruff + `terraform validate`. No unit/integration tests, no Azure SDK mocking.
- **Hardcoded ARM API version fallback table** — `_get_api_version()` maintains a static dict (e.g., `"microsoft.compute": "2024-03-01"`). Will drift as Azure deprecates API versions.
- **`shared_access_key_enabled = true`** — "break-glass" comment in code, but keys exist and can bypass the managed identity auth model if leaked.
- **Cost Management export removed from Terraform state** — `removed` block with `destroy = false`; must be created manually in Azure Portal (MCA billing-scope RBAC can't be managed via REST API for MSA-owned accounts).

### 4. Technical Challenges & Complex Workflows Resolved
1. **Dynamic provider management with backward-compatible legacy config migration (`app/main.py:_migrate_legacy_providers`, `app/caller.py:call`)** — Original config used flat SSM keys (`gemini_api_key`, `bridge_url`, `vertex_project_id`, etc.). New `providers` list supports arbitrary named providers with typed schemas. `_migrate_legacy_providers()` runs on every `GET /api/providers`, auto-promotes flat keys into the dynamic list, calls `cfg.patch_config()` to persist to SSM — idempotent, zero-downtime migration. `caller.py:call()` checks `cfg["providers"]` first, falls back to legacy flat keys second. `tools/migrate_vertex_to_proxy.py` handles the specific SSM rename of `vertex`/`gemini` provider values → `vertex-proxy`.

2. **MCP server via FastMCP StreamableHTTP transport (`app/mcp_server.py`, `app/main.py:23`)** — Three MCP tools (`chat`, `list_models`, `get_log`) exposed via FastMCP ≥2.0 `StreamableHTTP`, mounted on the same Uvicorn process at `/mcp` via `app.mount("/mcp", mcp.http_app(path="/"))`. Shares the process, port, and auth surface with the REST API. Tool implementations reuse `caller.call()` and `llm_log.*` exactly — consistent logging. The `chat` tool accepts `caller_name` + `purpose` annotations persisted to DynamoDB for MCP-vs-HTTP call traceability.

3. **Google Vertex AI per-request OAuth2 token refresh (`app/caller.py:_invoke_vertex`)** — Every Vertex call does a full `service_account.Credentials.from_service_account_info()` + `creds.refresh(GoogleAuthRequest())` on the synchronous request thread. SA credentials JSON stored in SSM, loaded per-request via `get_config()`. No token caching — each invocation makes one extra HTTPS round-trip to the Google OAuth2 endpoint. Region-aware URL: `"global"` → `aiplatform.googleapis.com`, otherwise `{region}-aiplatform.googleapis.com`.

### 5. Potential Bottlenecks & Technical Debt
- **No HA / SPOF** — Single EC2, no load balancer, no ASG. `systemctl restart seip-models` during deploy = downtime; instance loss = manual recovery required.
- **Per-request Vertex token refresh** — Google access tokens valid 3600 s; fetching on every call wastes 200–500 ms per Vertex request and hammers the token endpoint. Module-level cache with expiry check would eliminate this.
- **Synchronous DynamoDB write on the hot path** — `llm_log.update()` calls `_save_to_dynamo()` synchronously inside `POST /api/chat`, adding 5–20 ms DynamoDB write latency to every response. Should be a background task.
- **SSM `GetParameter` on every request** — `cfg.get_config()` called on every API operation with no in-process TTL cache; adds 10–30 ms RTT per call. SSM default rate limit: 40 TPS for `GetParameter`.
- **No auth on REST API / web UI** — FastAPI has no auth middleware. Relies entirely on Cloudflare Access. Any VPC host can call `http://dev-app-host:8184/api/chat` without credentials.
- **In-memory log deque is ephemeral** — `_log` deque (maxlen=500) is process-only; crash loses all in-flight "running" entries.

### 4. Technical Challenges & Complex Workflows Resolved
1. **Dual-role fck-nat bastion (NAT + DNAT + SSH gateway)** — Single `t3.micro` simultaneously acts as NAT gateway (`source_dest_check=false` + route `0.0.0.0/0 → NIC`), SSH jump host, and DNAT relay for two ports. iptables rules injected post-boot via Terraform `null_resource` provisioner: polls for SSM Online status (up to 4 min), runs `AWS-RunShellScript` to add PREROUTING/POSTROUTING rules, persists as systemd oneshot services (`seip-mcp-fwd.service`, `seip-portal-fwd.service`). DNAT targets the bastion's **private** IP (not EIP) because AWS translates EIP → private IP before iptables sees the packet.

2. **EKS outside Terraform state + orchestrated cleanup via API endpoints** — `module.eks_cluster` is commented out in `main.tf`; deployed via `./scripts/deploy_eks.sh` to decouple EKS lifecycle from VPC/ASG Terraform state and prevent accidental `terraform destroy` cascades. The `seip-orchestrator` FastAPI app exposes `/api/eks/cleanup/scaled-objects`, `/api/eks/cleanup/nodepools`, `/api/eks/cleanup/nodeclaims`, `/api/eks/cleanup/nodes` — execute `kubectl delete` sequences in safe order before targeted `terraform destroy -target=module.eks_cluster`. EKS module code: Karpenter 1.3.3 with SQS interruption queue via EventBridge rules (Spot/rebalance/state-change), KEDA 2.16.0 with IRSA.

3. **DynamoDB → PostgreSQL: claim mechanism redesign + KEDA trigger replacement** — AWS DynamoDB uses conditional `UpdateExpression` with `ConditionExpression="attribute_not_exists(analyzing_by)"` + `ClientError ConditionalCheckFailedException` retry loop for distributed worker claiming. PostgreSQL replaces this with `UPDATE events SET analyzing_by=... WHERE ... FOR UPDATE SKIP LOCKED RETURNING event_id` — single atomic statement, no retry loop. KEDA trigger changes from `aws-cloudwatch` (requiring deep-mind to `cloudwatch.put_metric_data()` every poll cycle + IAM IRSA + CloudWatch alarm infrastructure) to `postgresql` type querying `SELECT COUNT(*) FROM events WHERE analyzed=FALSE AND (analyzing_by IS NULL OR analyzing_at < NOW() - INTERVAL '120 seconds')` directly — removes the entire custom metric pipeline.

### 5. Potential Bottlenecks & Technical Debt
- **Single NAT/bastion SPOF** — All private subnet internet access, SSH, and DNAT routing depends on one `t3.micro` fck-nat. Stopped/replaced instance drops all outbound connectivity and loses DNAT rules (systemd services survive reboots but not instance replacements). No HA NAT, no NAT Gateway redundancy.
- **EKS outside Terraform state** — `terraform plan` cannot detect drift in the EKS cluster, Karpenter NodePools, or KEDA ScaledObjects. Helm provider must be manually uncommented in sync with the deploy script. Two-step bootstrapping required (`-target=aws_eks_cluster.main` then full apply) to avoid provider chicken-and-egg.
- **DynamoDB → PostgreSQL cutover risk** — In-flight events partially claimed by AWS-side workers during cutover won't be visible on Azure side — risk of duplicate or dropped analysis. No dual-write or blue/green cutover procedure documented beyond a single-step `INSERT INTO events SELECT ...` migration script.
- **Multi-repo submodule sync complexity** — ~10 git submodules with independent CI pipelines and independent `master` branches. `commit-all.sh`/`check-pull.sh` partially automate sync, but divergent HEADs, stale `.gitmodules` refs, and cross-service API contracts (e.g., seip-deep-mind → `http://seip-models:8184`) tracked only by convention; no integration or contract tests.
- **sentence-transformers cold-start** — Model loaded on first event batch (lazy, 384-dim). On KEDA scale-from-0, adds 30–60 s to first processing cycle. Model pre-baked into Dockerfile base image but must still be loaded into memory on each Pod start; not persisted across scale events.
- **Public S3 bucket for Lua filters** — `mysak7-seip-lua` is fully public (`s3:GetObject *`). KMS RSA-4096 signature verification is the sole integrity control; stale public key in `seip-agent-win/.env` causes silent bundle rejection.

## Project: az-llm-aks (Azure Spot LLM Orchestrator V2)
- **Business Domain:** LLM Inference Platform / Self-Hosted AI on Azure Spot Kubernetes
- **Core Architecture:** A FastAPI control-plane pod on a fixed system node pool manages the lifecycle of Ollama inference pods scheduled exclusively on Azure Spot CPU nodes provisioned on demand by Karpenter (AKS Node Auto-Provisioning). All Ollama pods share a single Azure Blob Storage container mounted via BlobFuse2 (`blob.csi.azure.com`) at `/root/.ollama/models`, eliminating per-node model downloads entirely.
- **Primary Tech Stack:** Python 3.12, FastAPI ≥0.111.0, uvicorn ≥0.29.0, httpx ≥0.27.0, Pydantic v2 ≥2.7.0, kubernetes ≥29.0.0, azure-cosmos ≥4.7.0, azure-identity ≥1.16.0, azure-storage-blob ≥12.19.0, azure-mgmt-compute ≥33.0.0, azure-monitor-opentelemetry ≥1.6.0; Terraform ≥1.7 / azurerm 4.69.0 / kubernetes provider 2.38.0 / helm ~2.17; Kubernetes 1.33 (AKS); Ollama inference runtime; Fluent Bit 3.1.

### 1. Infrastructure & Cloud Resources (PaaS/IaaS)

- **IaC Tool:** Terraform with split-state design. `terraform/base/` — permanent resources (RG, Storage Account, Blob Container, CosmosDB account+db+container, ACR, static public IP), state key `dev-base.terraform.tfstate`. `terraform/cluster/` — ephemeral resources (AKS, identity, federated credential, RBAC, ingress-nginx/cert-manager Helm releases, K8s ConfigMap/Secrets/ServiceAccounts/DaemonSet, monitoring module), state key `dev-cluster.terraform.tfstate`. ### 2. CI/CD & GitOps Automation
- **Pipelines:** Four GitHub Actions workflows:

### 4. Technical Challenges & Complex Workflows Resolved
1. **BASE/CLUSTER split state for nightly destroy safety** — Permanent resources (CosmosDB, Blob Storage, static IP, ACR) must survive nightly `terraform destroy`; AKS and K8s resources must be destroyable. Solution: two independent Terraform state files in the same backend. `cluster/resources.tf` contains `removed { lifecycle { destroy = false } }` blocks for every resource migrated out — without these, a subsequent `terraform apply` after cluster re-creation would attempt to adopt or destroy them. Two-stage bootstrap in `terraform-cluster.yml` (targeted apply → cluster ready check → full apply) required because `kubernetes` and `helm` providers need a live API server to initialise.

2. **BlobFuse2 dual PVC pattern (ReadOnlyMany + ReadWriteMany)** — Azure Blob CSI Driver supports both access modes via different `volumeHandle` identifiers (`ollama-models-handle` vs `ollama-models-handle-rw`) and `readOnly` flags. Inference Deployments from `generate_deployment.py` always reference `ollama-models-pvc` (ReadOnlyMany). Cache-pull Jobs in `api/routes/cache.py` reference `ollama-models-pvc-rw` (ReadWriteMany). One Job can populate the blob store with new model weights while existing inference pods continue serving; new inference pods immediately see the new weights on first mount — no per-node download, no PV resize.

3. **Streaming reverse proxy via httpx to in-cluster Ollama DNS (`api/routes/proxy.py`)** — Resolves models to `http://ollama-{safe_name}.default.svc.cluster.local:11434`. Performs fast `HEAD /api/tags` health check (5 s timeout) to distinguish "not provisioned" (ConnectError → 503) from "slow inference" (passes). Inspects request body for `"stream": false` to choose buffered (`Response`) vs streaming (`StreamingResponse`). Streaming path uses `httpx.AsyncClient.stream()` with `aiter_bytes()` — passes data chunks through without buffering, preserving Ollama's newline-delimited JSON token stream, 300 s timeout for long generation.

4. **Workload Identity federated credential across two Terraform states** — `identity.tf` in cluster layer creates user-assigned identity + `azurerm_federated_identity_credential` binding OIDC issuer URL (`azurerm_kubernetes_cluster.main.oidc_issuer_url`) to K8s subject `system:serviceaccount:default:control-plane-sa`. `azure.workload.identity/client-id` annotation on `control-plane-sa` ServiceAccount ties K8s → Azure identity. During nightly destroy/rebuild, federated credential re-created with the new cluster's OIDC URL — critical because AKS generates a new OIDC issuer per cluster instance.

### 5. Potential Bottlenecks & Technical Debt
- **CPU-only inference, GPU quota = 0** — Karpenter NodePool targets D/E/F/A/L CPU spot families (vCPU 2–4). All Ollama workloads on CPU; large models (≥13B parameters) are impractically slow. Adding NC-series requires quota increase + new NodePool tolerations + GPU device plugin.
- **Single-replica control plane, nightly destroy = zero SLA** — FastAPI Deployment uses `strategy: Recreate` (1 replica); any rollout or eviction = downtime. Cluster destroyed at midnight UTC; no inference availability until next manual/automated deploy. No auto-redeploy cron — midnight destroy is one-way.
- **Static single-IP whitelist, no authn** — `whitelist-source-range: "${ALLOWED_IP}/32"` accepts exactly one IP. Changing requires `git push` to `main`. No authentication layer (OIDC, API keys) — any request from the whitelisted IP can provision/deprovision models.
- **CosmosDB status never reconciled with K8s reality** — `status` field (`inactive|provisioning|running|error`) patched via `patch_item` but never reconciled with actual K8s Deployment state. A pod crash or Karpenter eviction leaves CosmosDB showing `running` while inference pod is gone.
- **Spot eviction = silent inference loss** — 1 replica per Ollama Deployment, no PodDisruptionBudget, no liveness probe, no HPA/KEDA. Karpenter consolidation (`consolidateAfter: 1m`) aggressively reclaims underutilised nodes. Eviction → pod restart → BlobFuse2 re-mount latency 30–90 s → 503s during that window. No fallback routing.
- **Secret management inconsistency** — `control-plane-secrets` K8s Secret synced by deploy workflow using raw `az cosmosdb keys list` (primary master key). Identity also has `Cosmos DB Account Reader Role` via Workload Identity. Two auth paths exist for CosmosDB data plane; SDK in `api/routes/models.py` uses the key from Secret, not the identity token — bypasses Workload Identity setup. Same issue for `azure-blob-secret` (raw storage account key for BlobFuse2 instead of CSI Workload Identity integration).
